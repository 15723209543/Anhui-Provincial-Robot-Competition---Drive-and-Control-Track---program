## 一、项目简介

本仓库整理的是我们参加 **2026 年安徽省驱动与控制竞赛 A 平台** 时使用和总结的一套代码资料。项目以主办方提供的例程为基础进行学习、拆分、重构和调试，并结合我们在备赛和决赛过程中遇到的实际问题，形成了一套更适合现场调试、模块复用和快速排错的代码包。

本项目对应的比赛结果为：**安徽省驱动与控制竞赛省一等奖**。

仓库主要内容来自我们自己整理的最终版资料，包括各器件的模块代码、测试工程、接线说明、调试总结和输出文档。我们希望把这份资料公开到 GitHub，帮助后续参加类似比赛、学习 STC8 单片机、调试驱动与控制平台的同学少走一些弯路。

## 二、上传动机

我们在备赛过程中发现，单片机竞赛真正困难的地方并不只是“代码能不能写出来”，而是以下几个问题：

1. 不同模块之间的引脚容易冲突，照搬例程经常会出现一个模块能跑、多个模块一起跑就出问题的情况。
2. 串口调试在比赛现场并不总是方便，尤其是在需要快速观察传感器、电机、舵机、键盘、灯带等状态时，外接显示会更直观。
3. 很多例程只证明“模块可以工作”，但没有整理成适合复用的 `.c` / `.h` 模块，也没有给出清楚的接线和正确现象说明。
4. 比赛现场时间紧，代码结构越清晰、测试现象越明确，越容易快速定位问题。

因此，我们把自己最终使用的资料整理出来，希望后来者不仅能看到代码，还能看到我们的整理思路、调试习惯和比赛经验。

## 三、与主办方例程相比的主要区别

本项目不是简单搬运主办方例程，而是在例程基础上做了大量适合比赛现场使用的整理和改造。

### 1. 全部统一使用 LCD12864 展示信息

这是本项目最重要的特点。

主办方例程中，不同模块可能采用串口、1602、数码管或其他方式观察结果。本项目中，我们尽量将所有模块的状态、测试结果和提示信息统一输出到 **LCD12864** 上。

这样做的好处是：

- 不依赖电脑串口助手，现场调试更方便；
- 传感器数据、键盘键值、时钟时间、电机状态、舵机角度等信息可以直接在屏幕上显示；
- 每个模块的测试现象更加直观，便于判断接线是否正确、程序是否运行；
- 统一显示入口，降低比赛现场排错成本。

例如：

- 4x4 矩阵键盘按键值显示到 LCD12864；
- 光敏传感器数据显示到 LCD12864；
- SHT30 温湿度数据显示到 LCD12864；
- 超声波测距结果显示到 LCD12864；
- DS1302 时间信息显示到 LCD12864；
- 直流电机当前测试状态显示到 LCD12864；
- WS2812 灯带当前模式和 RGB 值显示到 LCD12864；
- 蜂鸣器、LED、舵机等模块也配合 LCD12864 显示测试步骤或状态。

### 2. 按器件进行模块化封装

每个重要器件尽量单独整理为独立模块，包含：

- 模块 `.c` 文件；
- 模块 `.h` 文件；
- 测试工程代码；
- 接线说明；
- 正确测试现象说明。

这种结构更适合比赛前快速复习，也更方便在新工程中移植。

### 3. 统一头文件和基础类型

项目中统一使用：

```c
#include "STC8A8K64D4.H"
```

同时增加了 `common_type.h`，统一定义常用类型：

```c
typedef unsigned char u8;
typedef unsigned int  u16;
typedef unsigned long u32;
```

这样可以减少不同例程之间风格不一致的问题，也方便后续维护。

### 4. 增加大量接线说明和正确现象说明

很多模块不只是给出代码，还补充了接线方式、跳线设置、注意事项和“正常情况下应该看到什么现象”。

例如：

- LCD12864 说明了 J16 接口、相关跳线和灰度电位器；
- 超声波模块说明了 J23 接口中 GND、P71、P70、VCC 的对应关系；
- 直流电机说明了 P20 控制端、强制停止测试和 PWM 占空比测试；
- WS2812 灯带说明了 DI/DO 区别、共地要求、外部供电要求和测试模式。

这些说明对比赛现场排查问题非常有帮助。

### 5. 处理了部分实际调试中遇到的引脚冲突

在比赛平台上，多个模块共用或相邻接口时容易出现引脚冲突。本项目根据实际调试情况做了一些调整。

例如，WS2812 灯带原本容易与其他模块产生 IO 冲突，因此本版将其数据线调整到 **P1.0**，并在说明中明确写出修改原因、接线方法和代码修改位置。

### 6. 更重视“能在现场快速判断对错”

本项目中的很多 `main.c` 测试文件都加入了中文注释，说明每一步测试在做什么，以及正确现象是什么。这样在比赛现场不需要反复翻资料，只看屏幕、看模块现象就能大致判断程序和硬件是否正常。

## 五、文件夹外三个文件说明

除 `最重要器件/` 目录外，仓库外层还包含三个非常重要的资料文件。这三个文件并不是简单附带材料，而是我们在备赛、调试和比赛现场复用过程中整理出的核心参考资料。

### 1. `output.docx`：比赛现场可移植代码整理文档

`output.docx` 是我们将各模块关键代码、使用方法、测试说明和移植注意事项整理到一起形成的输出文档。它的定位不是普通说明书，而是一个可以在比赛现场快速查阅、快速复制、快速移植的代码资料。

在比赛现场，如果临时需要使用某个器件，或者原工程出现问题，可以通过该文档快速找到对应模块的核心代码、调用方式、注意事项和正确现象。它的价值主要体现在：

- 方便现场快速查找代码；
- 方便从单模块测试工程移植到综合工程；
- 方便在电脑、工程或文件夹混乱时快速恢复关键代码；
- 方便对照 LCD12864 显示内容判断模块是否正常；
- 适合作为比赛前复习和比赛现场应急参考资料。

简单来说，`output.docx` 是我们自己整理的“可带入比赛场地使用的代码移植手册”。

### 2. `展航科技-A平台-原理图.pdf`：硬件连接与引脚核对依据

`展航科技-A平台-原理图.pdf` 是 A 平台对应的原理图资料，主要用于确认接口、引脚、跳线、电源和模块之间的硬件连接关系。

在实际调试中，很多问题并不一定来自代码，而可能来自：

- 接口插错；
- 引脚理解错误；
- 跳线帽状态不对；
- 电源选择不对；
- 多个模块占用了同一组 IO；
- 某些外设与 LCD12864 或其他模块存在引脚复用。

因此，这份原理图是排查硬件问题的重要依据。尤其是在电机、舵机、超声波、LCD12864、LED、矩阵键盘等模块出现异常时，建议优先对照原理图确认接口和引脚。

### 3. `模块分析.xlsx`：我们自己总结的器件相关信息表

`模块分析.xlsx` 是我们自己在备赛过程中总结出来的器件分析表，也是整个仓库中非常重要的资料之一。它不是主办方直接给出的普通文件，而是我们根据例程、原理图、实物平台和实际调试情况整理出的模块信息表。

这个 Excel 表格主要用于记录和分析各个器件的关键信息，包括但不限于：

- 每个模块对应的接口位置；
- 每个器件使用到的单片机引脚；
- 模块所需的跳线帽或硬件设置；
- 模块使用的通信方式，例如普通 IO、I2C、单总线、定时器等；
- 模块是否依赖 `delay`、`lcd12864` 或其他公共代码；
- 模块之间可能存在的引脚冲突；
- 使用 LCD12864 显示时需要注意的兼容问题；
- 模块测试时的正确现象和异常排查方向。

这份表格的意义在于，它把分散在例程、原理图、代码和实物调试中的信息集中到了一起。对于比赛现场来说，查 Excel 往往比重新翻原理图、翻例程更快。尤其是在多个模块组合使用时，这个表可以帮助快速判断：

- 当前模块能不能和 LCD12864 同时使用；
- 当前模块是否占用了其他模块的 IO；
- 是否需要更换引脚或修改宏定义；
- 某个模块不工作时应该优先查硬件、查跳线还是查代码。

因此，`模块分析.xlsx` 可以理解为我们自己整理的“器件信息总表”和“比赛现场排错索引表”。

## 八、使用方法

### 1. 先测试 LCD12864

建议先打开 `1. LCD12864` 目录下的测试工程，确认 LCD12864 能正常显示。

如果 LCD12864 不显示，请优先检查：

- LCD12864 是否插在正确接口；
- J3 是否选择 5V；
- 相关跳线是否取下，避免数码管占用 LCD 引脚；
- RV2 灰度电位器是否需要调节；
- 下载时 IRC 是否选择 24MHz。

### 2. 再单独测试每个器件

不要一开始就把所有模块合到一个工程里。推荐按照下面顺序测试：

1. LCD12864；
2. LED / 蜂鸣器；
3. 矩阵键盘；
4. 光敏传感器；
5. SHT30；
6. 超声波；
7. DS1302；
8. AT24C02；
9. 舵机；
10. 直流电机；
11. WS2812 灯带。

每测试一个模块，都要先阅读该模块目录下的接线说明。

### 3. 移植模块时保留必要文件

大多数模块需要同时包含以下文件：

```text
STC8A8K64D4.H
common_type.h
delay.c / delay.h
lcd12864.c / lcd12864.h
对应模块的 .c / .h 文件
```

如果某个模块依赖定时器、中断或特定 IO，请同时检查对应的 `.h` 文件和 `main.c` 注释。

## 九、注意事项

1. 本项目主要用于学习、备赛和调试参考，不保证适用于所有硬件版本。
2. 不同批次平台的跳线、接口或丝印可能存在差异，使用前请结合自己的开发板原理图确认。
3. 电机、舵机、灯带等模块可能需要较大电流，必要时请使用外部 5V 电源，并确保单片机、模块和外部电源 **GND 共地**。
4. WS2812 灯带必须接 DI 输入端，不要误接 DO。
5. 如果出现显示异常、数据异常或模块不工作，请先单独测试 LCD12864 和对应模块，不建议直接在综合工程中排错。
6. 本项目部分代码来自对主办方例程的学习和改造，若仓库中包含主办方、厂商或第三方原始资料，请遵守其原始版权声明。

## 十、比赛心得与阶段感受

这次安徽省驱动与控制竞赛，对我们来说不是一次突然成功的比赛，而是一段从完全没底、不断试错、反复崩溃，到最后勉强稳住心态完成比赛的过程。现在回头看，最深的感受不是“我们有多强”，而是这一路真的太不容易了。很多时候并不是一直充满信心，反而经常觉得自己是不是做不下去了，是不是不该继续投时间进去，是不是准备了也没什么用。最后能拿到省一等奖，确实有努力，也有坚持，还有很多现场临时撑住的运气。

最开始的校赛阶段，因为第一次参加比赛没有任何经验，开局一股脑打模块代码，结果没有时间写main函数，好多模块打了也没有用上；再加上队友第一次配合不熟悉交流不到位，因为一个LCD12864的‘0’被打成‘\0’直接浪费大量时间；甚至因为对现场验收节奏也不熟悉，差点因为注释滚屏直接0分。这种不熟悉直接体现在成绩上，校赛最后只有 21 分，这个分数说实话挺打击人的，这是由于准备不够充分、对平台结构理解不深、队友之间交流配合有问题共同结果。而且当时比赛结束后第一反应不是马上总结，而是有点懵：明明前面也花了时间，为什么真正验收的时候就拿不到分？当时能明显感觉到，我们只是“照着例程抄”，校赛现场的节奏和自己平时慢慢调完全不一样，平时遇到问题可以慢慢查，还可以借助各种手段，但是比赛现场却要很快判断、很快改、很快验收。校赛之后心里其实有一点不服，也有一点慌，因为这个成绩说明我们距离真正能比赛还差很多。但它让我们第一次意识到：驱动与控制类比赛并不是把代码复制进去就结束，而是要真正理解硬件接口、模块依赖、引脚冲突、工程配置和现场调试流程。

从校赛到省初赛这段时间，我们的状态开始变得更紧。因为已经知道自己不是很稳，所以每次调模块的时候都会更在意细节。这个阶段最累的地方不是单纯写代码，而是不断重复“跑通、出错、修改、再记录”的过程。很多问题第一次遇到时会觉得莫名其妙，比如 LCD12864 显示不出来、传感器数据不对、电机有时候转有时候不转、定时器一改别的模块又受影响。开始时遇到这些问题会很烦，甚至会觉得是不是板子坏了，后来慢慢才接受：这类比赛本来就是这样，很多问题不会清清楚楚告诉你原因，只能一点一点试出来。

省初赛可以说是我们最舒服的一次。一方面比校赛暴露出来了很多问题让我们进步了；另一方面，在经过了一次比赛之后队友之间更加熟悉，配合更加默契。在整个比赛途中其实我们并没有出一些很大的问题，全程有条不紊地完成，最后拿到 45 分。

决赛前的一段时间反而是心理压力最大的时候。比赛和课程考试、日常作业挤在一起，时间并不宽裕。比赛那周甚至有线性代数考试，真的产生过不想继续参加的想法，那种感觉不是不想要成绩，而是觉得事情太多了，复习也要时间，比赛也要时间，继续准备又不一定有好结果，一边觉得已经走到决赛了，放弃很可惜；另一边又担心自己最后两边都做不好，现在想想，那一段最难的不是技术问题，而是心态上的摇摆。但是我们非常幸运的是遇到了一位伯乐，他给了我们一个很好解决办法，把我们从弃赛劝了回来。这里想给大家说一下，遇到问题真的不要第一反应弃赛，多想解决方法，等我们拿了省一之后回想真的很恐怖，但凡当时放弃了连省三都没有。

决赛当天的感受现在还很清楚。刚开始进场时其实很紧张，但还能控制住，因为前面已经准备了不少资料，心里想着至少可以按照之前整理的步骤一点点做。可是真正开始后，现场情况并没有想象中顺利。首先是一上来要两个电机两个超声波直接懵逼了，然后电脑突然自动关机，这种事情平时听起来只是小问题，但在比赛现场真的会让人脑子空一下。尤其是代码还在改、思路还没完全记下来时，电脑一黑，整个人会瞬间慌掉。那一刻不是简单的烦躁，而是会担心刚才改的东西是不是没了，时间是不是又被浪费了，后面还能不能赶回来。

比赛中后段，最折磨人的就是有些模块明明之前能跑，到现场却突然表现不对。特别是电机和风扇相关的问题，最后阶段真的让人非常崩溃。前面测试时能转，到了关键时刻就是不转，而且越接近结束越找不到原因。最后半个小时的时候，我们还在卡这个问题，当时心态已经非常紧张了。因为这个时候已经不是“慢慢查原因”的阶段，而是必须尽快让它跑起来。越着急越容易怀疑所有东西：是不是代码错了，是不是引脚错了，是不是供电错了，是不是工程配置错了，是不是之前能跑只是巧合。当时已经是最后几分钟。那个时候已经没有太多空间重新设计方案，只能抓住以前验证过能运行的部分，能复制就复制，能替换就替换，先保证现象出来。当时队友提醒可以直接尝试复制后面可以运行的代码，这个思路最后确实起了作用。说实话，那一刻并不是完全理解了所有原因之后的从容解决，更像是在巨大压力下抓住一个能用的办法。代码跑起来的时候，心里真的有一种劫后余生的感觉。虽然还会想“为什么这样就可以”，但比赛现场已经来不及把每个原因都彻底讲清楚，能在规定时间内把结果做出来，已经很不容易了。

比赛结束后，刚走出现场时并没有马上特别兴奋，更多是疲惫。前面几个月从寒假准备到校赛、省初赛、省决赛，很多晚上都在调代码、看资料、改工程，决赛当天又一直处在紧绷状态，所以结束的一瞬间反而有点空。后来知道获得省一等奖，才慢慢有一种不真实的开心。因为回想整个过程，我们并不是一路顺风，也不是每次都很自信，而是在很多次想放弃、很多次跑不通、很多次临时出问题之后，硬撑到了最后。这个结果对我们来说确实很珍贵。

这次比赛最让我感动的部分，是队友之间真的一起扛了下来。准备阶段有很多枯燥的工作，不是每次调试都有明显进展，有时候忙到很晚也只是解决一个小问题；比赛现场出问题时，两个人也会紧张，也会怀疑，但至少不是一个人在扛。一个人脑子乱的时候，另一个人还能提醒换个办法；一个人写代码的时候，另一个人还能看接线、查文件、找之前能跑的版本。最后这个结果不是某一个人的功劳，而是两个人在很多次不顺利的时候都没有彻底放弃。

现在把文件整理到 GitHub 上，也有一种把这段经历留下来的感觉。这个仓库里的代码和说明不一定完美，甚至有些地方以后看可能还会觉得不够规范，但是它确实记录了我们当时真实使用过、调试过、带到比赛现场准备移植的内容。`模块分析.xlsx` 不是随便放进去的表，而是那段时间一点一点总结出来的器件信息；`output.docx` 也不是普通说明文档，而是我们希望在现场能直接查、直接复制、直接移植的资料。它们背后其实对应着很多次试错、很多次返工和很多次临时救场。

如果只用一句话概括这次比赛，我觉得就是：过程比想象中难，结果比想象中好。校赛的失落、省初赛的紧张、决赛前的摇摆、决赛现场的崩溃和最后拿到结果的惊喜，全部混在一起，才构成了这次比赛完整的记忆。能拿到省一当然开心，但更重要的是，这段经历让我们真的经历了一次从不会、到敢做、到现场硬撑、到最后完成的过程。希望后来看到这个仓库的同学，不只是拿到一份代码，也能感受到这份资料背后那些真实的调试、压力、坚持和幸运。也祝愿后来参加安徽省驱动与控制竞赛以及其他电子设计类比赛的同学，都能顺利完成自己的作品，拿到让自己满意的结果。

## 十一、未完成与后续计划

当前版本为 **v2.4 最终版**，也是我们比赛时整理出的最终可用版本。但它并不是一个完全工程化、完全无缺陷的长期维护项目。

目前仍有一些没有完全完成或希望后续继续完善的内容：

- 部分功能没有完全实现；
- 部分模块仍以单独测试工程为主，综合工程还可以继续整理；
- 部分注释、接线说明和异常处理还可以更加规范；
- 后续可以继续补充更多比赛现场问题和解决方法；
- 后续可以进一步整理成更标准的 GitHub 开源项目结构。

欢迎后来者继续改进，也欢迎基于本项目整理出更完善的版本。
